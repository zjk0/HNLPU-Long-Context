# 关于复现HNLPU的思考

我是想基于HNLPU研究长上下文情景下的问题，但是我现在需要先复现出HNLPU这个平台。

把论文看完之后，我认为我需要复现的是硬件架构和数据流这两个部分，其中数据流部分更为重要。而Metal-Embedding部分则不用管。

kv cache的计算和attention compute cost的计算是相对容易写的，关键在于memory access latency和throughput的计算怎么搞。

在进行memory access latency的计算时，需要考虑kv cache有无溢出attention buffer，如果没有溢出，直接使用读取的kv cache的大小除以attention buffer的带宽就行，但是如果溢出了，溢出到HBM，这个时候就需要考虑HBM的带宽了，而且还要考虑是并行的读取attention buffer和HBM（二者同时读取）还是串行的读取attention buffer和HBM（先读一个，再读另一个）。

我发现，我的kv cache计算函数是计算全部kv cache的，也就是全部16个芯片的kv cache总和，但是在hnlpu的参数设置文件中，attention buffer的容量写的是单个chip上的attention buffer的容量。我觉得，如果是性能评估程序，计算kv cache还是计算全部kv cache的量会比较好。但是，这在memory_access_latency_eval中，会导致芯片中的kv cache有无溢出的判断变得可能没有想象中的简单。我想过将kv cache和attention buffer size * 16进行比较，但是这个比较我觉得不太行，因为我认为kv cache有无溢出的判断应该是要具体到各个芯片的，而不是直接比较总和。我现在是这样想，因为kv cache的计算是在列内进行的，各列是并行进行的，所以，各个列中的相同编号的芯片的kv cache应该是同步增加的，所以，各列中编号相同的芯片的kv cache的大小应该是相同的，可以基于这个特点来进行kv cache有无溢出的判断。
